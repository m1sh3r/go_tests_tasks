# Задачи для студентов по тестированию в Go
## Тема: Вычисление алгебраических выражений

Ниже представлены **6 вариантов заданий для каждого из 5 типов тестирования**. Все задачи связаны общей предметной областью (алгебраические вычисления), но могут выполняться независимо.

---

# 🔷 РАЗДЕЛ 1: ЮНИТ-ТЕСТЫ (6 вариантов)
*Фокус: базовые тесты с пакетом `testing`, проверка отдельных функций*

---

## 📋 Вариант 1.1: Базовые арифметические операции
**Задание:** Реализовать пакет `basicops` с функциями `Add`, `Subtract`, `Multiply`, `Divide`.
```go
func Add(a, b float64) float64
func Divide(a, b float64) (float64, error) // ошибка при делении на 0
```
**Требования к тестам:**
- Отдельная функция `TestXxx` на каждую операцию
- Проверка: деление на ноль, отрицательные числа, нулевые операнды
- Использовать `t.Errorf` с контекстом: `t.Errorf("Divide(10,0): expected error, got nil")`

---

## 📋 Вариант 1.2: Возведение в степень и корень
**Задание:** Реализовать `Power(base float64, exp int)` и `Sqrt(x float64)`.
**Требования к тестам:**
- Тесты на: `exp=0`, `exp<0`, `base=0`, `x<0` для корня
- Проверка точности `Sqrt`: `math.Abs(result-expected) < 1e-9`
- Тест паники/ошибки при некорректных входных данных

---

## 📋 Вариант 1.3: Вычисление факториала
**Задание:** Реализовать `Factorial(n int) (int64, error)`.
**Требования к тестам:**
- Граничные случаи: `n=0`, `n=1`, `n=20` (макс. для int64)
- Ошибка при `n<0` и `n>20` (переполнение)
- Проверка рекурсивной/итеративной реализации через одинаковые тесты

---

## 📋 Вариант 1.4: НОД и НОК чисел
**Задание:** Реализовать `GCD(a, b int) int` и `LCM(a, b int) int`.
**Требования к тестам:**
- Проверка: `GCD(0, x) = x`, `GCD(x, x) = x`
- Симметричность: `GCD(a,b) == GCD(b,a)`
- Связь НОД/НОК: `GCD(a,b) * LCM(a,b) == abs(a*b)`

---

## 📋 Вариант 1.5: Проверка простых чисел
**Задание:** Реализовать `IsPrime(n int) bool` и `NextPrime(n int) int`.
**Требования к тестам:**
- Тесты на: `n<2`, чётные числа, известные простые (2,3,5,7,11...)
- Проверка `NextPrime(10) == 11`, `NextPrime(13) == 17`
- Тест производительности для больших чисел (опционально)

---

## 📋 Вариант 1.6: Работа с дробями
**Задание:** Реализовать тип `Fraction` и методы `Add`, `Reduce`.
```go
type Fraction struct{ Num, Den int }
func (f Fraction) Add(other Fraction) Fraction
func (f Fraction) Reduce() Fraction // сокращение дроби
```
**Требования к тестам:**
- Проверка сложения: `1/2 + 1/3 = 5/6`
- Сокращение: `4/8 → 1/2`, отрицательные знаменатели
- Проверка инварианта: `Den > 0` после `Reduce()`

# 🔷 РАЗДЕЛ 2: ТАБЛИЧНЫЕ ТЕСТЫ (6 вариантов)
*Фокус: паттерн `[]struct{}`, субтесты `t.Run()`, массовая проверка сценариев*

---

## 📋 Вариант 2.1: Решение квадратного уравнения
**Задание:** `QuadraticRoots(a,b,c float64) ([]float64, error)`.
**Таблица тестов должна включать:**
| Сценарий | a | b | c | Ожидаемый результат |
|----------|---|---|---|---------------------|
| Два корня | 1 | -5 | 6 | [3, 2] |
| Один корень | 1 | -4 | 4 | [2] |
| Нет корней | 1 | 2 | 5 | [] |
| Линейное (a=0) | 0 | 2 | -4 | [2] |
| a=b=0, c≠0 | 0 | 0 | 5 | ошибка |

**Требования:** Использовать `t.Run(tc.name, ...)` и сравнение с допуском для `float64`.

---

## 📋 Вариант 2.2: Вычисление полинома (схема Горнера)
**Задание:** `Evaluate(coeffs []float64, x float64) float64`.
**Таблица тестов:**
```go
[]struct {
    name   string
    coeffs []float64  // [a0, a1, a2] = a0 + a1*x + a2*x²
    x      float64
    want   float64
}{
    {"константа", []float64{5}, 10, 5},
    {"линейный 2x+3", []float64{3,2}, 4, 11},
    {"x² при 1e6", []float64{0,0,1}, 1e6, 1e12},
    {"пустой срез", []float64{}, 5, 0}, // или ошибка
}
```

---

## 📋 Вариант 2.3: Вычисление дискриминанта и его анализ
**Задание:** `AnalyzeDiscriminant(a,b,c float64) (d float64, rootsCount int)`.
**Таблица тестов:** Проверка всех комбинаций знака дискриминанта:
- `D > 0` → 2 корня
- `D = 0` → 1 корень  
- `D < 0` → 0 действительных корней
- Граничные значения: `D = ±1e-15` (проверка на точность)

---

## 📋 Вариант 2.4: Классификация треугольника по сторонам
**Задание:** `TriangleType(a,b,c float64) (string, error)`.
**Таблица тестов:**
```go
[]struct {
    sides    [3]float64
    wantType string  // "equilateral", "isosceles", "scalene", "impossible"
    wantErr  bool
}{
    {[3]float64{3,3,3}, "equilateral", false},
    {[3]float64{3,3,5}, "isosceles", false},
    {[3]float64{1,2,10}, "impossible", true}, // неравенство треугольника
}
```

---

## 📋 Вариант 2.5: Конвертер систем счисления
**Задание:** `ToBase(n int, base int) (string, error)` — перевод в 2..36 систему.
**Таблица тестов:**
- `ToBase(10, 2) = "1010"`, `ToBase(255, 16) = "ff"`
- Ошибки: `base < 2`, `base > 36`, `n < 0`
- Граничные: `n=0`, максимальное int для разных баз

---

## 📋 Вариант 2.6: Валидация алгебраического выражения
**Задание:** `ValidateExpr(expr string) error` — проверка синтаксиса `"ax²+bx+c"`.
**Таблица тестов:**
```go
[]struct {
    expr    string
    valid   bool
    desc    string
}{
    {"x² + 2x + 1", true, "полное выражение"},
    {"2x²-3x", true, "без константы"},
    {"x³ + 1", false, "степень > 2"},
    {"2x + + 3", false, "двойной оператор"},
    {"", false, "пустая строка"},
}
```

# 🔷 РАЗДЕЛ 3: МОКИ С GOMOCK (6 вариантов)
*Фокус: изоляция зависимостей, генерация моков через `mockgen`*

---

## 📋 Вариант 3.1: Сервис с кэшем коэффициентов
**Интерфейс:**
```go
type Cache interface {
    Get(key string) (string, bool)
    Set(key, value string, ttl time.Duration)
}
```
**Задание:** Реализовать `CachedSolver`, который сначала проверяет кэш, затем вычисляет.
**Тест с моком:**
- `EXPECT().Get("eq1").Return("1,-3,2", true)` → решение из кэша
- `EXPECT().Get("eq2").Return("", false)` → вычисление + `Set()`
- Проверка, что `Set` вызван ровно 1 раз при промахе кэша

---

## 📋 Вариант 3.2: Сервис с внешним API коэффициентов
**Интерфейс:**
```go
type APIClient interface {
    FetchCoeffs(eqID string) (a,b,c float64, err error)
}
```
**Задание:** `RemoteSolver` использует клиент для получения данных.
**Тест с моком:**
- Успешный ответ от API → корректное решение
- Ошибка сети (`err != nil`) → проксирование ошибки пользователю
- Таймаут: `EXPECT().FetchCoeffs(...).After(100*time.Millisecond)`

---

## 📋 Вариант 3.3: Логгер и аудит вычислений
**Интерфейс:**
```go
type Logger interface {
    Info(msg string, fields map[string]any)
    Error(msg string, err error)
}
```
**Задание:** `AuditedSolver` логирует каждое вычисление.
**Тест с моком:**
- `EXPECT().Info(gomock.Any(), gomock.Any()).AnyTimes()`
- Проверка, что при ошибке вызван `Error()`, при успехе — `Info()`
- Использование `gomock.InOrder()` для проверки последовательности

---

## 📋 Вариант 3.4: Хранилище истории решений
**Интерфейс:**
```go
type HistoryStore interface {
    Save(eqID string, roots []float64) error
    GetHistory(limit int) ([]Record, error)
}
```
**Задание:** `PersistentSolver` сохраняет результаты после вычисления.
**Тест с моком:**
- `Save()` возвращает ошибку → решение всё равно возвращается, но логируется предупреждение
- Проверка, что `Save` вызывается с корректными аргументами: `gomock.Eq("eq1"), gomock.Len(2)`

---

## 📋 Вариант 3.5: Валидатор входных данных
**Интерфейс:**
```go
type Validator interface {
    ValidateCoeffs(a,b,c float64) error
}
```
**Задание:** `ValidatingSolver` проверяет коэффициенты перед решением.
**Тест с моком:**
- `EXPECT().ValidateCoeffs(0,0,5).Return(err)` → ранний возврат ошибки
- `EXPECT().ValidateCoeffs(1,-3,2).Return(nil)` → продолжение вычисления
- Тест композиции: мок валидатора + мок хранилища в одном тесте

---

## 📋 Вариант 3.6: Фабрика решателей по типу уравнения
**Интерфейс:**
```go
type SolverFactory interface {
    Create(eqType string) (Solver, error)
}
type Solver interface {
    Solve(coeffs []float64) ([]float64, error)
}
```
**Задание:** `Router` выбирает решатель через фабрику.
**Тест с моком:**
- Мок фабрики возвращает мок-решатель для `"quadratic"`
- Проверка, что `Solve()` вызван с правильными коэффициентами
- Ошибка фабрики (`Create("cubic")`) → корректная обработка

# 🔷 РАЗДЕЛ 4: GOLDEN FILES (6 вариантов)

---

---

## 📋 Вариант 4.2: Генерация отчёта о решении

**Требования:**

---

## 📋 Вариант 4.4: Генерация уравнения по коэффициентам (LaTeX)
**Задание:** `ToLaTeX(a,b,c float64) string` → `x^{2} - 3x + 2 = 0`.
**Golden-файлы:**

## 📋 Вариант 4.5: Вывод пошагового решения
**Задание:** `StepByStepSolution(a,b,c float64) string` — объяснение для студента.

- Тест на порядок строк (сортировка по ID)

# 🔷 РАЗДЕЛ 5: FUZZING (6 вариантов)
*Фокус: встроенный fuzzing Go 1.18+, поиск краевых случаев, устойчивость к некорректному вводу*

---

## 📋 Вариант 5.1: Фаззинг парсера выражений
**Задание:** `ParseExpr(s string) (a,b,c float64, err error)`.
**Fuzz-тест:**
```go
func FuzzParseExpr(f *testing.F) {
    f.Add("x² + 2x + 1")
    f.Add("2x²-3x+4")
    f.Fuzz(func(t *testing.T, s string) {
        a, b, c, err := ParseExpr(s)
        if err == nil {
            requireFinite(t, a, b, c) // нет NaN/Inf
        }
        // Главное: нет паник при любом вводе
    })
}
```

---

## 📋 Вариант 5.2: Фаззинг функции возведения в степень
**Задание:** `Power(base float64, exp int) float64`.
**Fuzz-тест:**
- Проверка: результат конечен (`!math.IsInf`, `!math.IsNaN`)
- При `exp >= 0` и `base == 0` → результат 0 (не паника)
- Седе: `0, 1, -1, 2, -2, 10, -10` для экспоненты

---

## 📋 Вариант 5.3: Фаззинг валидации коэффициентов
**Задание:** `Validate(a,b,c float64) error`.
**Fuzz-тест:**
```go
func FuzzValidate(f *testing.F) {
    f.Fuzz(func(t *testing.T, a,b,c float64) {
        err := Validate(a,b,c)
        // Функция не должна паниковать ни при каких float64
        // Ошибка — допустимый результат для некорректных данных
    })
}
```
**Цель:** Найти комбинации, вызывающие панику (деление на 0, недопустимые операции).

---

## 📋 Вариант 5.4: Фаззинг конвертера строк в числа
**Задание:** `ParseFloatSafe(s string) (float64, error)` — безопасный парсинг.
**Fuzz-тест:**
- Сравнение с `strconv.ParseFloat`: если стандартная функция не паникует, наша тоже
- Проверка обработки: пустая строка, пробелы, `"NaN"`, `"Inf"`, очень длинные строки
- Седе: валидные числа, нечисловые строки, граничные форматы

---

## 📋 Вариант 5.5: Фаззинг функции округления
**Задание:** `RoundTo(x float64, precision int) float64`.
**Fuzz-тест:**
- Проверка: результат отличается от исходного не более чем на `0.5 * 10^-precision`
- Обработка: `precision < 0`, очень большие `precision`, `x = ±Inf, NaN`
- Цель: найти случаи, где округление даёт неожиданный результат

---

## 📋 Вариант 5.6: Фаззинг генератора случайных уравнений
**Задание:** `GenerateRandomEq(seed int64) (a,b,c float64)`.
**Fuzz-тест:**
```go
func FuzzGenerateRandomEq(f *testing.F) {
    f.Fuzz(func(t *testing.T, seed int64) {
        a, b, c := GenerateRandomEq(seed)
        // Проверка инвариантов:
        // - хотя бы один коэффициент ≠ 0
        // - все коэффициенты конечны
        // - детерминизм: одинаковый seed → одинаковый результат
    })
}
```
**Особенность:** Проверка детерминизма через повторный вызов с тем же seed.

# 📦 Общие рекомендации для всех вариантов

## Структура проекта
```
algebra-testing/
├── unit/          # Варианты 1.1–1.6
├── tabledriven/   # Варианты 2.1–2.6
├── mocks/         # Варианты 3.1–3.6 (+ mocks/)

└── README.md      # Инструкция по запуску каждого варианта
```

## Полезные команды
```bash
# Юнит-тесты
go test ./unit/... -v -cover

# Табличные тесты с фильтрацией
go test ./tabledriven/... -run TestQuadraticRoots/два_корня -v

# Моки: генерация
mockgen -source=interface.go -destination=mocks/mock_interface.go -package=mocks

# Golden files: обновление эталонов
go test ./golden/... -update -v

# Fuzzing: запуск на 30 секунд
go test ./fuzzing/... -fuzz=FuzzParseExpr -fuzztime=30s

# Проверка на гонки данных (для всех)
go test -race ./...

# Генерация отчёта о покрытии
go test ./... -coverprofile=coverage.out
go tool cover -html=coverage.out -o coverage.html
```
